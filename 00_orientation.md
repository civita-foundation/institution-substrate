# Institution Substrate  
## Orientation & Guardrails

---

## Purpose of This Project

This project formalizes **Institution Substrate**:  
the underlying science and engineering required to understand, design, and operate **executable institutions**.

It exists to answer a single core question:

> *Why do institutions fail even when intentions, talent, and resources are present — and what would it take to make them operationally reliable?*

This is not a policy document.  
This is not an ideology.  
This is not a reform playbook.

It is a **substrate**: the minimum conceptual and structural layer required before any institutional intervention can be responsibly attempted.

---

## What This Project Is

This project is:

- A **formal ontology** of institutions  
- A **feasibility-first** framework (constraints precede ideals)
- A bridge from:
  - institutional theory  
  - to executable, testable systems
- A unification of:
  - governance
  - trust
  - coordination
  - time
  - execution

It treats institutions as **systems**, not narratives.

---

## What This Project Is Not

This project is explicitly **not**:

- A political ideology
- A moral theory
- A governance manifesto
- A consulting slide deck
- A training curriculum
- An “AI transformation” framework

Any attempt to apply this work without feasibility analysis is considered misuse.

---

## Core Design Principle

> **Desirability does not imply feasibility.**  
> **Good intentions do not override constraints.**

Institutions fail not because people are bad,  
but because **constraints are violated**.

This project starts with constraints.

---

## Science vs Engineering (Strict Separation)

This project is deliberately split into two layers:

### 1. Institution Science
Addresses:
- What institutions *are*
- How trust emerges or collapses
- Why coordination fails
- Why copying institutions does not work
- How time, memory, and drift operate

This layer makes **truth claims**.

---

### 2. Institution Engineering
Addresses:
- What can be formalized
- What must remain human
- How rules become executable
- How institutions operate across time
- How errors are detected and contained

This layer makes **design commitments**.

Science precedes engineering.  
Engineering without science is malpractice.

---

## Why “Substrate”

The term *substrate* is used deliberately.

Just as:
- software runs on an OS
- biology runs on chemistry
- governance runs on institutions

So too:
> institutions run on a deeper substrate of constraints, trust, and coordination logic.

Most institutional theories operate **above** this layer.  
This project operates **below** it.

## How to Read This Project

The files are structured to be read **sequentially**, but each file is also **self-contained**.

Recommended path:
1. Ontology (what exists)
2. Feasibility (what is possible)
3. Computation (what can be formalized)
4. Execution (what can run)
5. Time & drift (what survives)
6. Scale & governance (what generalizes)

Skipping feasibility will invalidate downstream conclusions.
